## 一、問題陈述 (Problem Statement)

台灣保險市場長年存在三大結構性痛點,讓一般民眾在投保與理賠兩個階段都處於高度資訊不對稱的位置:

1. **保單散落難以管理**:多數保戶手上同時持有壽險、醫療險、意外險、產險等不同公司保單,紙本與 PDF 散落在抽屉、Email、Line 群組,出險時往往找不到自己有哪些保障、也不知道保障額度與到期日。
2. **條款複雜、無法檯向比較**:各家保險公司的除外責任、等待期、給付條件、理賠文件要求皆不同,保戶難以用一致標準評估「哪一張保單對我最好」,業務員亦難以提供中立比較。
3. **理賠流程黑箱**:出險後民眾常不知道第一步該做什麼——要通知誰、要準備什麼文件、要多久能拿到錢——導致錯過時效或漏報項目,理賠權益受損。銀髮族群子女更是經常需要代為處理,跨世代資訊傳遞斷裂。

這些問題不是單一保戶的偶發狀況,而是制度性、結構性的。根據金融消費評議中心統計,保險類申訴常年高居前三名,其中超過六成與「理賠認定」與「條款理解落差」相關。本專案目標是把上述三件事變得「集中、可比、透明」。

## 二、解決方案 (Solution)

打造一個**純前端 SPA (Single Page Application)**,以「保險地圖」概念提供三大主軸功能:

- **保單集中管理**:使用者可在瀏覽器本地新增、編輯、刪除自己持有的保單,系統自動彙整到期日、保額、保費、險種分類,並用儀表板呈現「總保障額度」「年繳保費」「即將到期提醒」三大指標。
- **各家條款對比**:預載台灣 30 家保險公司 (含壽險、產險) 的標準條款資料,使用者可選 2-4 張保單並排比較除外責任、等待期、住院日額、癌症給付等欄位,並對差異項目高亮標示。
- **理賠 SOP 透明化**:依險種 (住院、手術、癌症、失能、身故、意外) 提供 step-by-step 理賠流程,包含「通知期限」「應備文件清單」「常見補件原因」「預估工作天數」,並允許使用者勾選已完成項目以跟蹤進度。

技術上採**純前端 + localStorage** 架構,所有個資與保單資料只存在使用者瀏覽器,不上傳任何伺服器,兼顧隱私與法規 (個資法) 要求。

## 三、功能清單 (Feature List)

### 3.1 保單管理 (Policy Vault)
- 新增/編輯/刪除保單,欄位含:保險公司、保單號碼、險種、被保險人、要保人、保額、保費、繳費期間、保障期間、續保狀態。
- 自動分類:壽險、醫療險、意外險、癌症險、住院日額、長照、失能、旅平險、車險、產險。
- 到期提醒儀表板:30 天內即將到期保單以紅色標示。
- 匯出 JSON 備份 / 匯入 JSON 還原。
- 一鍵列印保單清單 (適合銀髮族子女攝帶紙本回老家確認)。

### 3.2 條款對比 (Comparison Matrix)
- 預載 30 家保險公司基礎條款資料 (公開資訊,不含個資)。
- 並排比較 2-4 張保單,支援險種維度:住院日額、癌症一次金、化療/放療給付、重大器官移植、豁免保費、等待期、除外責任。
- 差異高亮 (紅/綠標記 best/worst),並顯示「這張保單在 X 項目優於其他 N 張」摘要。
- 條款來源標註 (金管會公開資料、保發中心、公司官網),點擊可查看原始連結。

### 3.3 理賠 SOP (Claim Workflow)
- 6 種險種 SOP:住院理賠、手術理賠、癌症理賠、失能理賠、身故理賠、意外醫療。
- 每種 SOP 含 4 階段:**通報 (T+0)** -> **文件準備 (T+3)** -> **送件 (T+7)** -> **核付 (T+14~30)**。
- 應備文件清單含 check box,完成項目自動存進 localStorage。
- 「常見補件原因」提示:如「診斷書未記載 ICD-10 碼」「收據影本未蓋醫院章」等實戰經驗彙整。
- 預估工作天數表 (依公司別),並提供「理賠權益試算」:輸入自付額試算可申請金額。

### 3.4 台灣保險地圖 (Insurance Map)
- 視覺化圖表:全台 30 家保險公司總部位置、業務密度、申訴率 (來自金管會公開資料)。
- 篩選器:依險種、依地區、依申訴率高低。

### 3.5 通用功能
- 響應式設計 (RWD):桌機、平板、手機皆可使用,字級可放大供銀髮族閱讀。
- 深色模式切換。
- 完整繁體中文 UI,符合台灣用語習慣。
- 不需登入、不需註冊、不需安裝 App。

## 四、技術棣 (Tech Stack)

| 層次 | 技術 | 理由 |
|------|------|------|
| 前端框架 | React 18 + Vite | 快速建置、社群資源豐富、HMR 開發體驗佳 |
| 路由 | React Router v6 | SPA 多頁面導航 (保單/對比/理賠/地圖) |
| 狀態管理 | Zustand | 比 Redux 輕量,適合純前端本地狀態 |
| 樣式 | Tailwind CSS + Headless UI | 快速 RWD、無障礙組件支援 |
| 資料儲存 | localStorage (含 IndexedDB 備援) | 純前端、零後端、個資不外流 |
| 圖表 | Recharts | 保險地圖、儀表板圖表 |
| 預載資料 | 靜態 JSON (`/public/data/companies.json`) | 30 家公司條款資料 |
| 部署 | Vercel | 自動 CI/CD、全球 CDN、免費 HTTPS |
| 版本控制 | GitHub (`openclawsean024-create/insurance-hub`) | 公開原始碼、利於社群協作 |
| 程式碼品質 | ESLint + Prettier | 一致風格 |

**非技術決策**:
- **不串接金管會 API**:目前金管會無開放保單資料查詢 API,預載資料採手動維護。
- **不做後端**:個資保護優先,所有資料本地儲存。
- **不做 OCR 辨詘保單**:初期僅手動輸入,MVP 完成後再評估。

## 六、風險與決策 (Risks & Decisions)

### 6.1 個資與法規風險
- **風險**:雖然資料存於 localStorage 不上傳,但若使用者公用電腦仍可能漏洞。
- **決策**:UI 上明顯提示「資料儲存在此裝置,清除瀏覽器資料將遺失」;匯出/匯入 JSON 機制讓使用者自備備份;不採用指紋、臉部等生物辨詘。

### 6.2 條款資料正確性風險
- **風險**:保險條款會修訂,預載資料可能過期造成誤導。
- **決策**:每筆條款標註「資料更新日期」,並在 UI 顯示「本資料僅供參考,實際理賠以保險公司公告為準」免責聲明。

### 6.3 法律責任風險
- **風險**:若使用者因本工具決策而受損,是否需負責?
- **決策**:在首頁與所有條款頁頂部固定顯示免責聲明;本工具定位為「資訊整理」而非「保險建議」;不提供任何「應該買哪家」的具體推薦。

### 6.4 維護成本風險
- **風險**:30 家保險公司資料需定期更新,若無社群協作會過勞。
- **決策**:開源 GitHub 接受社群 PR;建立資料更新 checklist;預估每月維護工時 4-6 小時。

### 6.5 變現衝突風險
- **風險**:若未來導入業務員導流,會與「中立比較」定位衝突。
- **決策**:初期不做導流;若未來考慮,須明確標示「業配」並維持免費中立版完整可用。

## 七、變現路徑 (Monetization Path)

本專案短期以**建立信任與使用者基礎**為優先,變現採三階段漸進:

### Phase 1 (0-12 個月):完全免費,累積使用者
- 無廣告、無付費牆、無個資萃集。
- 透過 SEO 內容行銷 (例如「2026 醫療險比較」「癌症險怎麼挑」) 導入自然流量。
- 目標:12 個月內達 5 萬 MAU、GitHub 100 star。

### Phase 2 (12-24 個月):B2B 經紀人工具訂閱
- 推出「保險經紀人版」:可同時管理 50 位客戶保單、產生客製化比較報表 PDF、客戶端白牌版本。
- 訂價:月繳 NT$499 / 年繳 NT$4,990。
- 目標:24 個月內 200 位付費經紀人 = NT$100 萬 ARR。

### Phase 3 (24 個月+):資料授權與 API
- 與產險/壽險公司合作,提供「民眾投保前參考資料」授權。
- 開放資料 API 給金融科技新創,採用量計價。
- 評估俅購或策略投資出場。

**核心原則**:任何變現模式都不能牺牲「中立、透明、免費核心版完整」三大承諾。免費版永遠免費、資料永遠公開透明、不賣使用者個資。
